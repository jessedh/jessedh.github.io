About T3Token

What is T3Token?

T3Token is an ERC20-compliant stablecoin built on the Ethereum blockchain that introduces revolutionary features to enhance transaction security and user control.

Unlike traditional cryptocurrencies, T3Token implements a unique "HalfLife" mechanism that creates a time-based constraint on token transfers, allowing for transaction reversals within a specified period.

This innovative approach addresses key challenges in digital transactions, including fraud prevention, error correction, and trust building between parties.

Key Innovations

  • Time-based transaction constraints
  • Reversible transactions
  • Adaptive security based on transaction patterns
  • Trust-building mechanics
  • Full ERC20 compatibility

How It Works

Initial Transfer

Sender initiates a token transfer to the recipient. The transaction is recorded on the blockchain with metadata including the HalfLife period.

HalfLife Period

During this period (default: 1 hour), the recipient cannot forward the tokens to another address, but can reverse the transaction back to the sender.

Commitment

After the HalfLife period expires, the transaction is considered final. The recipient can now freely transfer the tokens to other addresses.

Reversal Process

  1. Within the HalfLife period, either the sender or recipient can initiate a reversal.
  2. The reversal function is called with the recipient's address, the sender's address, and the exact amount.
  3. The contract verifies that the reversal is within the HalfLife period and that the tokens are being returned to the original sender.
  4. If all conditions are met, the tokens are transferred back to the sender and the transfer metadata is cleared.

Adaptive HalfLife

Trust Building

As transaction count between the same parties increases, the minimum HalfLife duration decreases.

High-Value Protection

Transactions significantly larger than the average receive an extended HalfLife period.

Pattern Reset

After 30 days of inactivity, the rolling average resets, restoring higher security for new transactions.

Use Cases

Enhanced Security

The reversal capability provides protection against errors, fraud, and unauthorized transactions. If a user accidentally sends tokens to the wrong address or a transaction is made without proper authorization, it can be reversed within the HalfLife period.

Example:

A company accidentally sends a payment to the wrong vendor. With T3Token, they can reverse the transaction within the HalfLife period, preventing financial loss.

Trust Building

New trading partners can start with longer HalfLife periods that automatically reduce as trust builds through repeated transactions. This creates a natural progression from high security to higher convenience.

Example:

A marketplace platform uses T3Token for transactions between buyers and sellers. New users have longer HalfLife periods, but as they build a transaction history, the periods automatically shorten for more efficient trading.

High-Value Protection

Unusual or high-value transactions automatically receive extended protection through longer HalfLife periods, providing additional security when it matters most.

Example:

A real estate transaction using T3Token would automatically receive an extended HalfLife period due to the high value, giving both parties more time to verify all aspects of the transaction before it becomes final.

Fraud Prevention

The cooling period prevents immediate re-spending of received tokens, reducing certain types of fraud and allowing time for suspicious activities to be detected and reversed.

Example:

A fraudster attempts to quickly move stolen tokens through multiple accounts. With T3Token, the HalfLife period prevents immediate forwarding, giving the victim time to detect and reverse the initial fraudulent transaction.

Key Functions

Overrides the standard ERC20 transfer function to include metadata tracking.

function transfer(address recipient, uint256 amount) public override returns (bool) {
    _transfer(_msgSender(), recipient, amount);
    transferData[recipient] = TransferMetadata({
        commitWindowEnd: block.timestamp + halfLifeDuration,
        halfLifeDuration: halfLifeDuration,
        originator: _msgSender(),
        transferCount: transferData[recipient].transferCount + 1,
        reversalHash: keccak256(abi.encodePacked(_msgSender(), recipient, amount))
    });
    return true;
}

Allows for transaction reversals within the HalfLife period.

function reverseTransfer(address from, address to, uint256 amount) external {
    TransferMetadata memory meta = transferData[from];

    require(msg.sender == from || msg.sender == to, "Only sender or receiver can reverse");
    require(block.timestamp < meta.commitWindowEnd, "HalfLife expired");
    require(to == meta.originator, "Reversal must go back to originator");
    require(balanceOf(from) >= amount, "Insufficient balance to reverse");

    _transfer(from, to, amount);

    // Prevent reversal of reversal (no metadata created)
    delete transferData[from];
}

Advanced transfer function with adaptive HalfLife duration based on transaction patterns.

function transferWithHalfLife(address to, uint256 amount, uint256 requestedHalfLife) public returns (bool) {
    // Calculate decay-based minimum
    uint256 txCount = transactionCountBetween[msg.sender][to];
    uint256 decayMin = 43200 >> txCount;

    // Load rolling average
    RollingAverage storage avg = rollingAvg[msg.sender][to];
    uint256 average = avg.count > 0 ? avg.totalAmount / avg.count : 0;

    // Reset if 30 days passed
    if (block.timestamp > avg.lastUpdated + 30 days) {
        avg.totalAmount = 0;
        avg.count = 0;
    }

    // Update rolling average
    avg.totalAmount += amount;
    avg.count += 1;
    avg.lastUpdated = block.timestamp;

    // Check for high-value transaction boost
    uint256 highValueMin = (average > 0 && amount > (average * 150) / 100) ? 3600 : 0;

    // Determine applied HalfLife
    uint256 finalMin = decayMin > highValueMin ? decayMin : highValueMin;
    uint256 appliedHalfLife = requestedHalfLife > finalMin ? requestedHalfLife : finalMin;

    // Execute transfer
    _transfer(msg.sender, to, amount);

    // Set transfer metadata
    transferData[to] = TransferMetadata({
        commitWindowEnd: block.timestamp + appliedHalfLife,
        halfLifeDuration: appliedHalfLife,
        originator: msg.sender,
        transferCount: transferData[to].transferCount + 1,
        reversalHash: keccak256(abi.encodePacked(msg.sender, to, amount, block.timestamp))
    });

    transactionCountBetween[msg.sender][to]++;
    return true;
}

Frequently Asked Questions

The HalfLife mechanism is a time-based constraint on token transfers that creates a "cooling period" during which transactions can be reversed. By default, this period is set to 3600 seconds (1 hour), but it can adapt based on transaction patterns between parties.

Transactions can only be reversed under specific conditions:

  • The reversal must occur within the HalfLife period
  • Only the sender or receiver can initiate a reversal
  • Tokens must be returned to the original sender
  • The full amount must be reversed

Additionally, once a transaction has been reversed, that reversal cannot itself be reversed.

The adaptive HalfLife adjusts the duration of the reversal window based on several factors:

  • Transaction history between the parties (more transactions = shorter HalfLife)
  • Transaction amount relative to average (higher than usual = longer HalfLife)
  • User-requested duration (must be at least the minimum calculated value)

This system automatically provides higher security for new relationships and unusual transactions while offering convenience for established patterns.
